# ❓ Почему не работает?

## Вы видите эти ошибки:

```
params: a1929434-a0c0-4521-a4ab-051d129d4b6e,1
JWTSessionError
GET /api/admin/stats 401
```

---

## Причина 1: SQL скрипт НЕ выполнен

### Симптомы:
- Ошибки продолжаются
- Пользователь `a1929434...` все еще делает запросы

### Как проверить:
Выполните в Supabase SQL Editor:
```sql
SELECT * FROM users WHERE id = 'a1929434-a0c0-4521-a4ab-051d129d4b6e';
```

**Если вернулась 1 строка** - SQL НЕ выполнен!  
**Если вернулось 0 строк** - SQL выполнен ✅

### Решение:
1. Откройте: https://supabase.com/dashboard/project/sjxepisvuthynvixpwii/sql
2. Скопируйте содержимое файла `ПОСЛЕДНИЙ_ШАНС.sql`
3. Вставьте в SQL Editor
4. Нажмите RUN
5. Дождитесь "Success"

---

## Причина 2: Cookies НЕ очищены

### Симптомы:
- SQL выполнен (пользователь удален)
- Но ошибки продолжаются
- Браузер отправляет старый токен

### Как проверить:
1. F12 → Application → Cookies → http://localhost:3000
2. Посмотрите на список cookies

**Если есть cookies** - они НЕ очищены!  
**Если пусто** - cookies очищены ✅

### Решение:
1. F12 → Application → Cookies
2. Кликните на http://localhost:3000
3. Нажмите "Clear all site data" (или иконку корзины)
4. Закройте браузер полностью
5. Откройте заново

---

## Причина 3: Сервер НЕ перезапущен

### Симптомы:
- SQL выполнен
- Cookies очищены
- Но ошибки продолжаются
- Кеш Next.js содержит старые данные

### Как проверить:
Посмотрите в терминал где запущен `npm run dev`

**Если сервер запущен давно** - НЕ перезапущен!  
**Если только что запустили** - перезапущен ✅

### Решение:
1. В терминале нажмите Ctrl+C
2. Дождитесь остановки
3. Введите: `npm run dev`
4. Дождитесь "✓ Ready in..."

---

## Причина 4: Браузер НЕ закрыт

### Симптомы:
- Все сделано
- Но ошибки продолжаются
- Кеш браузера не очищен

### Как проверить:
Вы закрывали браузер ПОЛНОСТЬЮ?

**Если только вкладку** - НЕ закрыт!  
**Если весь браузер** - закрыт ✅

### Решение:
1. Закройте ВСЕ вкладки
2. Закройте браузер (Alt+F4 или крестик)
3. Подождите 5 секунд
4. Откройте браузер заново

---

## Причина 5: Используете старую вкладку

### Симптомы:
- Все сделано правильно
- Но в ОДНОЙ вкладке ошибки
- В новой вкладке работает

### Как проверить:
Откройте новую вкладку (Ctrl+T) и перейдите на http://localhost:3000/admin/login

**Если в новой вкладке работает** - проблема в старой вкладке!  
**Если и в новой не работает** - проблема не в этом

### Решение:
1. Закройте ВСЕ вкладки с localhost:3000
2. Откройте НОВУЮ вкладку
3. Перейдите на http://localhost:3000/admin/login

---

## Причина 6: NextAuth кеширует сессию

### Симптомы:
- SQL выполнен
- Cookies очищены
- Сервер перезапущен
- Но ошибки продолжаются

### Как проверить:
Попробуйте режим инкогнито (Ctrl+Shift+N)

**Если в инкогнито работает** - проблема в кеше!  
**Если и в инкогнито не работает** - проблема глубже

### Решение:
1. Откройте режим инкогнито (Ctrl+Shift+N)
2. Перейдите: http://localhost:3000/admin/login
3. Войдите: admin@example.com / admin123

Если работает - очистите кеш браузера:
1. Ctrl+Shift+Delete
2. Выберите "Все время"
3. Отметьте "Cookies" и "Кеш"
4. Нажмите "Очистить данные"

---

## Причина 7: .env.local не загружен

### Симптомы:
- Ошибки подключения к БД
- "Connection refused"
- "Invalid credentials"

### Как проверить:
Посмотрите в терминал при запуске:
```
- Environments: .env.local, .env
```

**Если нет .env.local** - файл не загружен!  
**Если есть** - файл загружен ✅

### Решение:
1. Проверьте что файл `.env.local` существует
2. Проверьте что в нем правильные данные
3. Перезапустите сервер

---

## Причина 8: Неправильный DATABASE_URL

### Симптомы:
- "Connection timeout"
- "Connection refused"
- Долгие таймауты (15+ секунд)

### Как проверить:
Откройте `.env.local` и проверьте `DATABASE_URL`

**Должно быть:**
```
DATABASE_URL=postgres://postgres.sjxepisvuthynvixpwii:7gWa6KSOYpPTSanP@aws-1-eu-central-1.pooler.supabase.com:6543/postgres?sslmode=require&pgbouncer=true
```

### Решение:
Скопируйте правильный URL из Supabase:
1. https://supabase.com/dashboard/project/sjxepisvuthynvixpwii/settings/database
2. Connection string → Connection pooling
3. Скопируйте и вставьте в `.env.local`

## Диагностика: Что проверить

### 1. Проверьте пользователя в БД:
```sql
SELECT * FROM users WHERE id = 'a1929434-a0c0-4521-a4ab-051d129d4b6e';
```
**Должно быть:** 0 rows

### 2. Проверьте нового администратора:
```sql
SELECT id, email, role FROM users WHERE email = 'admin@example.com';
```
**Должно быть:** 1 row с НОВЫМ UUID

### 3. Проверьте сессии:
```sql
SELECT COUNT(*) FROM sessions;
```
**Должно быть:** 0 или только новые сессии

### 4. Проверьте cookies:
F12 → Application → Cookies → http://localhost:3000
**Должно быть:** Пусто или только новые cookies

### 5. Проверьте логи сервера:
Не должно быть:
- `a1929434-a0c0-4521-a4ab-051d129d4b6e`
- `JWTSessionError`
- `401` или `403`
